糟糕代码示例一：头文件常见问题

  • iOS

1. 在public头文件中import非必要的头文件

如果.h文件的定义、声明不依赖于另一个头文件，则不应import它；如果.m文件对它有依赖，应在.m文件中对其进行import.

2. 模块内交叉依赖的对象，交叉import；

应当以@class, @protocol 的方式，先进行类型声明；不要交叉import.

3. 错误地在头文件中使用static声明静态变量

一般不在头文件中使用static来声明静态变量。static 声明变量时，表明该变量的作用域在模块内(.o文件)有效。 如果在头文件中声明， 所有import它的.m文件，实际上会对应有不同的变量实例。

4. 在头文件中声明非公开方法、非protected字段

一个类的公开的头文件主要包含以下内容：

  • 公开的：公开给使用者调用的方法、属性；对其他类的类簇扩展方法（公开）；
  • 受保护的：提供给子类访问的保护字段（读／写）；子类可以重写的保护方法；
  • 以上方法、属性的依赖项（类型定义、协议定义、宏定义、常量定义等）

以下内容不宜在公开的头文件中定义：

  • 模块内使用的受保护方法（java inner访问属性）、依赖项定义等，应当在另外一个.h文件中定义（类扩展等）；
  • .m文件自己使用的私有字段、方法、常量等， 应当在.m文件开始时进行定义（类扩展、常量定义等）；
  • 其他组件相关的类簇方法，应由其他组件的头文件声明，不应由当前类提供。

5. 引入不必要的类型依赖

如果可以使用基类对象，不应引入子类。

例如，如果用UIViewController可以，不需要使用ChildViewController 子类；

6. 命名冲突

需要恰当的使用前缀命名；
